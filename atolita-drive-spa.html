Core Principles for Google Drive Organization

  • Centralization: Use Shared Drives for team-owned content as they are owned by the team, not an individual. This prevents data loss if an employee leaves.
  • Logical Hierarchy: Create a clear, intuitive folder structure that mirrors your company's departments and projects.
  • Granular Permissions: Apply the "principle of least privilege," giving users only the access they need to perform their tasks.
  • Consistency: Maintain consistent naming conventions and structures across all drives and folders.
  • Dedicated External Sharing: Create specific, controlled areas for sharing with external parties.
  • Regular Review: Periodically review your structure and permissions to ensure they remain relevant and secure.

Access & Permissions Strategy

Effective permission management is crucial for data security and proper collaboration. Google Shared Drives offer robust controls. This section outlines the recommended approach to assigning access levels for Shared Drives, folders, and individual files.

1. Shared Drive Level Permissions

These roles are assigned at the highest level of a Shared Drive and determine overall access capabilities:

Manager Role

Can manage members, content, and settings of the Shared Drive. Typically for Department Heads or IT Admins.

Content Manager Role

Can add, edit, move, and delete content. Suited for Team Leads or Senior Staff managing content.

Contributor Role

Can add and edit content. The standard role for most team members actively working within the drive.

Commenter Role

Can only comment on files. Useful for cross-functional collaborators needing to provide feedback.

Viewer Role

Can only view files. Appropriate for general company access to resources or for users who only need to read information.

General Rule: Grant a team "Content Manager" or "Contributor" access to their own departmental Shared Drive. Grant "Viewer" or "Commenter" access to other relevant Shared Drives where they need visibility but not editing rights.

2. Folder Level Permissions (within Shared Drives)

  • Inheritance: Permissions are typically inherited from the parent Shared Drive or folder.
  • Override: You can restrict permissions at the folder level. For example, a confidential sub-folder within a department's drive might have more limited access than the drive itself.
  • External Sharing: When sharing with external parties, ALWAYS use the dedicated `Atolita - External Collaboration` Shared Drive. Share only specific folders or files within it, granting the minimum necessary permissions (e.g., Viewer, Commenter). Avoid sharing entire departmental Shared Drives externally.

3. Individual File Level Permissions

  • Use sparingly for one-off shares or specific document collaborations when folder-level permissions are not suitable.
  • Expiration Dates: For external shares, always set an expiration date for access if the collaboration is temporary to maintain security.

Workflow Integration & Implementation

Adopting this new structure requires clear communication and consistent practices. This section covers key workflow integrations, naming conventions, and a step-by-step guide for implementation.

Workflow & Naming Conventions

  • File Naming: Encourage consistent file naming for easy identification and searching. A good practice is: YYYYMMDD_ProjectName_DocumentTitle_Version.ext (e.g., 20250521_Q2Campaign_MarketingPlan_V2.docx)
  • Folder Naming: Use clear, descriptive names for folders. Retain the numbering for top-level departmental drives (e.g., `01_Executive`, `02_Production`) as it aids in sorting and quick navigation.
  • "My Drive" Usage: Encourage employees to use their personal "My Drive" only for:
    • Truly personal, non-company-related files.
    • Temporary working files or drafts that will eventually be moved to the appropriate Shared Drive once finalized or ready for collaboration. Company-owned work should reside in Shared Drives.
